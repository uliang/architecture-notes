§ 01 — The same shape, three scales

One pattern, recurring at every distance.

Software architecture has fewer ideas than it lets on. The same shape — a thin intermediary sitting between a noisy outside and a structured inside — appears at three scales of distance, wearing three different names, and people argue them as if they were unrelated.

An API Gateway sits between many clients and many services across a network. It routes, it authenticates, it rate-limits. The internals don't know who's calling; the callers don't know who's answering.

A Backend for Frontend sits between one client and the same services. It exists because a mobile app and a desktop web app want different shapes of the same data, and forcing one API to serve both produces a shape that serves neither.

An Application Layer — call it use cases, application services, interactors — sits between presentation code and bounded contexts inside the same process. No network, no TLS, no rate limiter. But the same job: translate the outside's shape into the inside's shape, and back again.

Three names. One discipline. A thin band of routing & translation, then the real work.

Same shape, different distance. The discipline doesn't change — the things you protect against do.— On recurring patterns

The shared failure mode is also the same at every scale: business logic creeps inward, the thin layer fattens, and what was supposed to be a translator becomes a parallel domain — anaemic on one side, redundant on the other. The cure is the same too. Keep the layer routing-shaped, not reasoning-shaped.

§ 02 — Flask as presentation, only

How thin can a Flask app actually be?

Thin enough that grepping the Flask package for sqlalchemy returns nothing. Thin enough that the view functions read like dispatch tables. Thin enough that deleting Flask would leave the rest of the application intact, untestably-coupled to nothing, runnable from a CLI.

A view should do four things, in this order: parse the HTTP request into a typed command, hand the command to an application service, receive a plain data object back, and render it. No SQLAlchemy session, no transactions, no domain rules, no model imports. A handler is five to fifteen lines.

@bp.post("/students/<student_id>/enroll")
def enroll_student(student_id):
    command = EnrollStudentCommand(
        student_id=student_id,
        course_id=request.form["course_id"],
        term=request.form["term"],
    )
    try:
        result = services.enrollment.enroll(command)
    except CourseFullError as e:
        return render_template("partials/error.html", message=str(e)), 409
    return render_template("partials/enrollment.html", enrollment=result)

Notice what isn't there. No db.session. No .commit(). No model import. No transaction boundary. Flask doesn't know SQLAlchemy is on the path. The view's only outside contact is a function call on services.enrollment — an object that arrived already constructed, and could just as well have been a mock.

↑ Stays in Flask

  • request parsing (form, json, query)
  • routing & URL → handler
  • response shape (html, htmx, json)

§ 03 — The composition root

Where the database URL does live.

If Flask doesn't know about the database, and the domain doesn't know about the database, who does? A third thing, with no other job: the composition root. It reads the environment, builds the engine, builds the repositories, builds the application services, and hands the finished objects to whoever needs them. Flask is one such recipient. A CLI is another. A worker is another.

This is the inversion that makes the whole thing work. Configuration doesn't flow into the framework — services flow out of the wiring code, and the framework receives them already-constructed. The framework is downstream of configuration, not the source of it.

One wiring point. Many consumers. The framework is downstream of configuration, not the source of it.

The same services object reaches Flask, a Click CLI, a background worker, and an MCP server. None of them imports the database URL. None of them reaches for os.environ. They each accept whatever build_services() hands them and get on with their job.

@dataclass
class CoreConfig:
    database_url: str

    @classmethod
    def from_env(cls) -> "CoreConfig":
        return cls(database_url=os.environ["DATABASE_URL"])

def build_services(config: CoreConfig) -> Services:
    engine = create_engine(config.database_url, pool_pre_ping=True)
    session_factory = sessionmaker(bind=engine, expire_on_commit=False)
    uow_factory = lambda: SqlAlchemyUnitOfWork(session_factory)
    return Services(
        enrollment=EnrollmentService(uow_factory),
        payments=PaymentService(uow_factory),
    )
# the composition root
core_config = CoreConfig.from_env()
services    = build_services(core_config)
app         = create_app(services)

§ 04 — Litmus tests

How you know it worked.

A design like this is easy to describe and harder to maintain. Persistence has a way of seeping back upward — a quick query in a view, a session attached to flask.g, an ORM model returned to a template that lazy-loads on a relationship attribute. Each looks innocent. Each is a return to coupling.

So you carry a few tests in your pocket, the kind you can run mentally before merging, and occasionally for real.

  • TEST 01 Grep the Flask package for "sqlalchemy". Finds nothing. Not as an import, not as a string. If it appears, persistence has leaked into the presentation tier.
  • TEST 02 Run alembic upgrade head with Flask uninstalled. Migrations belong to the core, not the web framework. If they need Flask on the path, the boundary is decorative, not real.
  • TEST 03 Write a 20-line CLI that calls an application service. It should be possible without importing a single thing from the flask_app package. If the CLI starts pulling in blueprints or request contexts, the core isn't really standalone.
  • TEST 04 Delete the Flask package. Do core tests still pass? They should. Application service tests don't run a web server — they construct a service, pass a command, assert on the returned DTO. If they break, the test was secretly testing the framework.
  • TEST 05 What gets returned from an application service? A dataclass, a Pydantic model, a plain tuple. Never a SQLAlchemy ORM object — those carry a live session and will lazy-load against a closed connection somewhere in a template. Fully materialised DTOs cross the boundary.
  • TEST 06 Could you swap Flask for FastAPI in a weekend? Not as a real plan — as a thought experiment. If the answer is yes, you have a presentation adapter. If the answer is "we'd have to rewrite the data access too," you have something else, and it isn't separated.

Each of these is a small ritual. None is exotic. Together, they keep the discipline that the three diagrams above were promising. The thin layer stays thin only because someone keeps checking.

Coda

A single discipline, scaled.

An API Gateway protects services from clients. A BFF protects services from one specific client's appetite. An Application Layer protects bounded contexts from views. The artefacts differ — Envoy versus a GraphQL BFF versus a folder of .py files called application/ — but the work is the same: translate, orchestrate, don't decide.

The reward, when you keep it, is small and quiet: the inside of the application stops caring about the outside. You can swap Flask for FastAPI. You can run a use case from a CLI. You can mount an MCP server on the same core and let an agent drive it. The thing that was load-bearing — the domain — was never tangled with the thing that called it.

That's the entire pattern. A thin layer, at every scale, refusing to do the inside's job.
